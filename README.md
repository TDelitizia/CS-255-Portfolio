# CS-255-Portfolio
Portfolio repository for CS255 System Analysis and Design work

**Project Summary**

The client, DriverPass, is a company that provides driving instruction and training services to students preparing for their driver's license exams. 
DriverPass needed a web based system that would allow students to:

Schedule driving lesson appointments
Purchase and manage learning packages
Access online training materials
Make payments securely

The system also needed to support instructors and administrative staff by allowing them to:

Manage appointments
Track student progress
Maintain schedules
Process payments
Generate reports

This project required creating a Business Requirements document and a System Requirements document. Together, these documents demonstrate the full system
analysis and design lifecycle -- from gathering requirements to developing UML modes and  technical specifications. 

**What I did well**

One of my strongest areas in this project was clearly defining the organizing requirements. I carefullly separated functional and nonfunctional requirements
and ensured that each requirement was written as an atomic, testable statement using proper "The system shall..." language. I also demonstrated strong understanding
of UML modeling by developing:

A Use Case Diagram
Two Activity Diagrams
A Sequence Diagram
A Class Diagram

These models aligned directly with the requirements and reflected the system's structure, behavior, and interactions between users and components. Additionally, I improved
significantly in writing documentation that was structured, professional, and client-focused rather than technical only. 

**If I Could Revise One Part**

If I could revise one portion of my work, I would refine the Business Requirements Documents further by tightening the clarity and precision of some requirement statements. 
While they were functional and testable, I could improve them by ensuring even stronger traceability between stakeholder needs and system features. I would also enhance
diagram consistency by reviewing naming conventions across all UML artifacts to ensure perfect alignment between documents. 

**Interpreting User Needs**

To interpret the user's needs, I carefully analyzed the DriverPass interview transcript and identified core problems the client was trying to solve, such as manual scheduling
inefficiencies, lack of only access to training materials, and difficulty tracking student progress. I translated those needs into:

Functional requirements (scheduling appointments, processing payments)
Nonfunctional requirements (system performance, security, usability, availability)

Considering user needs is critical because a system that does not align with stakeholder expectations will fail, even if it is technically sound. Systems must solve real business
problems and support real users.

**My Approach to Designing Software**

When designing software, I follow a structured approach:
Gather and clarify requirements from stakeholders.
Separate functional and non functional requirements.
Model system behavior using UML diagrams.
Ensure traceability between requirements and design artifacts.
Validate that the system supports business goals.

In the future, I will continue using techniques such as:
Use case analysis
Activity and sequence modeling
Iterative requirement refinement
Stakeholder validation checkpoints

This project strengthened my understanding of how system analysis directly impacts system success. It reinforced that strong requirements and clear design documentation are the 
foundation of effective software development.
